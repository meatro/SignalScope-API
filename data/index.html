A DBC is a dictionary. It turns a location such as “CAN 0x321, first byte” into a useful name such as OilTemperature.

What happens when I load one?

The firmware validates and stores it as /dbc/active.dbc. A DBC swap clears subscriptions and mutation rules so old bit locations cannot silently affect a different database.

An oil-temperature display

  1. Search the catalog for the temperature signal.
  2. Confirm it starts near ambient and rises gradually.
  3. Copy its catalog index while prototyping.
  4. Edit data/index.html and fetch that value.

That useful app did not require a CAN driver, bit decoder, task scheduler, gateway, recorder, or custom protocol. SignalScope already owns those pieces.

data/index.html
const index = 42;

async function updateOilTemperature() {
  const response = await fetch(
    `/api/signal_catalog?indexes=${index}`
  );
  const { signals } = await response.json();
  const oil = signals[0];

  oilTemperature.textContent = oil?.live
    ? `${oil.value.toFixed(0)} °C`
    : "--";
}

setInterval(updateOilTemperature, 500);

Runtime indexes are convenient while experimenting. A finished app should persist the stable pair CAN ID + signal name, because indexes can change after loading a different DBC.
